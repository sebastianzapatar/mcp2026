Idea central

Dos problemas, dos soluciones

Para que los estudiantes no mezclen conceptos, separa la clase en tres capas: uv organiza el proyecto Python; pipx instala aplicaciones de terminal en entornos aislados; MCP organiza la comunicación entre IA y herramientas.

uv

Entorno reproducible

Declara dependencias, genera lockfile, crea el entorno y ejecuta comandos.

pipx

Herramientas aisladas

Instala aplicaciones CLI sin ensuciar el Python global ni el entorno del proyecto.

MCP

Protocolo de herramientas

Le dice al modelo qué herramientas existen, qué parámetros reciben y cómo llamarlas.

Parte 1 · uv

uv no es solo un pip más rápido

La idea clave para los estudiantes: uv administra el ciclo completo de un proyecto Python. Crea el entorno, resuelve dependencias, congela versiones, ejecuta comandos y ayuda a empaquetar.

Analogía para clase

pip instala paquetes; uv se parece más a un “director de obra”: mira los planos del proyecto, consigue los materiales exactos, prepara el taller y ejecuta el trabajo en el entorno correcto.

Instalación

Paquetes

Agrega o quita librerías con uv add y uv remove, sin editar archivos críticos a mano.

Entorno

.venv automático

Puede crear y usar el entorno virtual del proyecto, evitando el clásico “en mi computador sí funciona”.

Reproducibilidad

uv.lock

Guarda versiones exactas del árbol completo de dependencias para que todos instalen lo mismo.

Ejecución

uv run

Corre comandos dentro del entorno correcto, incluso si el estudiante olvidó activar el virtualenv.

Instalación clara

Cómo instalar uv sin confundirlo con pip

La recomendación didáctica es instalar uv como una herramienta de terminal, no como una librería del proyecto. Así queda disponible como comando global: uv.

Frase para clase

“pip instala paquetes dentro de un Python; uv es una herramienta que administra proyectos Python completos”.

pip · pipx · uvx

Por qué usar pipx en lugar de pip para herramientas

La confusión más común es usar pip install para todo. Para librerías de un proyecto está bien; para aplicaciones de terminal es mejor aislarlas con pipx.

Instalar una CLI con pip global

  • Puede mezclar herramientas con librerías del proyecto.
  • Puede generar conflictos entre versiones.
  • A veces requiere permisos del sistema o modifica el Python global.
  • Es más difícil saber qué paquete instaló qué comando.
  • Ejemplo a evitar: pip install uv en el Python global.

Instalar una CLI con pipx

  • Crea un entorno virtual independiente para cada aplicación.
  • Expone solo el comando en el PATH.
  • No contamina el entorno del proyecto.
  • Permite actualizar o desinstalar la herramienta de forma limpia.
  • Ejemplo válido: pipx install uv.
pip install requests

Úsalo dentro de un entorno/proyecto cuando quieres instalar una librería importable.

pipx install uv

Úsalo cuando quieres instalar una aplicación de terminal y mantenerla aislada.

uvx ruff check .

Úsalo para ejecutar una herramienta una sola vez, en un entorno temporal.

uv add mcp

Úsalo dentro del proyecto para declarar una dependencia del código.

El problema real

Por qué requirements.txt se queda corto

Un requirements.txt puede servir para proyectos simples, pero en clase conviene mostrar sus límites: no describe el proyecto completo, no separa claramente dependencias directas/transitivas y no resuelve el problema de las herramientas globales. Ahí entran uv para el proyecto y pipx para las aplicaciones de terminal.

Pregunta para activar la clase

“Si yo instalo hoy y ustedes instalan dentro de dos meses, ¿podemos estar seguros de que tendremos exactamente las mismas versiones?”

Flujo clásico

  • python -m venv .venv: crear entorno.
  • source .venv/bin/activate: activarlo manualmente.
  • pip install: instalar paquetes.
  • pip freeze: exportar todo, mezclando paquetes principales y paquetes secundarios.
  • Problema: es fácil olvidar pasos o subir un archivo desactualizado.

Flujo con uv

  • uv init: crea estructura del proyecto.
  • uv add: declara dependencia directa.
  • uv lock: resuelve árbol completo.
  • uv sync: instala exactamente lo bloqueado.
  • uv run: ejecuta usando el entorno correcto.
Archivos clave

Qué papel cumple cada archivo

Esta diapositiva evita una confusión común: pyproject.toml no cumple el mismo papel que uv.lock. Uno declara intención; el otro congela una resolución exacta.

pyproject.toml

El contrato editable del proyecto

Nombre, versión, versión mínima de Python, dependencias directas y scripts/entry points. Este archivo sí se revisa y se entiende.

uv.lock

La foto exacta del entorno

Versiones concretas, dependencias transitivas y datos de resolución. Normalmente no se edita a mano; uv lo actualiza.

.venv/

El entorno local

Carpeta generada en cada máquina. No se sube a Git. Se puede borrar y reconstruir con uv sync.

.python-version

La versión de Python esperada

Ayuda a que todos trabajen con una versión compatible, por ejemplo Python 3.12.

Movimiento claro

Cómo se mueve uv dentro del proyecto

El punto amarillo muestra el recorrido didáctico: una decisión humana entra al proyecto, uv resuelve el árbol de dependencias, sincroniza el entorno y ejecuta el servidor.

Instala el entorno local exactamente según el lockfile.

Corre weather-mcp sin activar nada a mano.

Terminalsecuencia para explicar en vivo
uv init weather-mcp          # crea un proyecto Python moderno
uv add "mcp[cli]>=1.28.0"   # agrega MCP como dependencia directa

Comandos esenciales

Qué comando usar según la situación

Para estudiantes, conviene enseñarlo por intención y no como lista suelta de comandos. Cada comando responde a una pregunta concreta del desarrollador.

Regla simple

Si vas a cambiar dependencias del proyecto, usa uv add/remove. Si vas a reconstruir el entorno, usa uv sync. Si vas a ejecutar el proyecto, usa uv run. Si vas a instalar una herramienta global de Python, usa pipx o el instalador oficial.

uv init

Crear un proyecto desde cero con estructura moderna.

uv add paquete

Agregar una dependencia directa y actualizar el lockfile.

uv add --dev pytest

Agregar herramientas solo para desarrollo o pruebas.

uv remove paquete

Quitar una dependencia y recalcular el entorno.

uv sync

Reconstruir el entorno exactamente como dice uv.lock.

uv run comando

Ejecutar dentro del entorno correcto sin activarlo manualmente.

uvx herramienta

Probar una CLI de Python en entorno temporal sin instalarla globalmente.

uv build

Construir el paquete distribuible cuando se quiere publicar.

pipx install uv

Instala uv como aplicación aislada cuando no usas el instalador oficial ni Homebrew.

pipx list

Muestra qué aplicaciones de terminal están instaladas con pipx y en qué entornos viven.

Parte 2 · MCP

MCP es el USB-C de las herramientas de IA

MCP es un estándar abierto para que una aplicación de IA descubra y use capacidades externas de forma ordenada. La palabra clave es protocolo: no es una API específica, ni un modelo, ni una base de datos.

Frase para clase

“MCP no le da inteligencia al modelo; le da un camino ordenado para usar capacidades externas”.

Modelo

Razona y decide

Interpreta la pregunta del usuario y decide si necesita una herramienta.

Host

Orquesta la experiencia

Claude Code, Codex u otra app muestra la conversación, conecta servidores y gestiona permisos.

MCP

Protocolo común

Define mensajes, capacidades, tools, recursos, prompts y transporte.

Servidor

Integra el mundo externo

Implementa la lógica: consulta Open-Meteo, valida datos y devuelve una respuesta estructurada.

El problema M×N

Sin MCP, todo se vuelve integración a mano

Si hay muchos clientes de IA y muchas herramientas, escribir conectores uno por uno se vuelve costoso. MCP propone que cada herramienta hable un estándar común.

Actores del protocolo

Host, cliente y servidor no son lo mismo

Esta distinción hace que MCP se entienda mucho mejor. En una exposición, dedica un minuto solo a separar estos tres roles.

Host

La aplicación de IA

Es donde vive la experiencia del usuario: Claude Code, Codex, un IDE o una app propia. Decide qué servidores se conectan y cómo se muestra la respuesta.

Cliente MCP

El conector interno

Es la pieza dentro del host que mantiene una sesión con el servidor MCP, negocia capacidades y envía solicitudes.

Servidor MCP

La herramienta externa

Es el programa que tú escribes. Expone tools, recursos o prompts; en este caso, weather-mcp expone get_weather.

Qué puede exponer un servidor

MCP no es solo tools

Para esta demo usamos una tool, pero MCP puede organizar varios tipos de contexto. Esto ayuda a que los estudiantes no reduzcan MCP a “llamar funciones”.

ToolsResourcesPromptsTransportsCapabilities
Tools acciones

Funciones que el modelo puede pedir ejecutar, como get_weather(latitude, longitude).

Resources datos

Información que el servidor puede ofrecer, como archivos, registros, documentación o resultados de consulta.

Prompts plantillas

Flujos o instrucciones reutilizables que ayudan a ejecutar tareas frecuentes con estructura.

Transports canal

La forma como viajan los mensajes. En esta demo local usamos stdio; en otros casos puede haber transporte remoto.

Comparación didáctica

MCP no reemplaza la API: la envuelve

La API externa sigue existiendo. MCP agrega una capa reusable, validada y entendible para el modelo. Así el modelo no inventa URLs, no maneja credenciales y no interpreta respuestas crudas.

Contrato tipadoEl modelo ve nombre, descripción, parámetros y tipos esperados.
Validación fuera del promptEl servidor valida latitud y longitud antes de llamar a Open-Meteo.
ReutilizaciónClaude Code, Codex u otro host usan el mismo servidor sin reescribir la integración.
Control y permisosEl usuario decide qué servidores conecta; el host puede pedir aprobación antes de acciones sensibles.

Esta es la explicación paso a paso para que el movimiento de MCP quede claro. El modelo no “sale a internet”; pide ejecutar una herramienta que vive en un servidor.

El host lee la configuración

Encuentra que existe un servidor llamado weather-mcp.

Arranca el servidor

El cliente MCP pregunta qué tools, recursos o prompts están disponibles.

El modelo decide llamar

Con la pregunta del usuario, el modelo elige get_weather y propone parámetros.

El servidor ejecuta

Valida entrada, consulta Open-Meteo y devuelve un resultado estructurado.

def get_weather(latitude: float, longitude: float, elevation: float | None = None):
    if not -90 <= latitude <= 90:
        raise ValueError("latitude must be between -90 and 90")
    if not -180 <= longitude <= 180:
        raise ValueError("longitude must be between -180 and 180")
    return _fetch_weather(latitude, longitude, elevation)

Exponerlo en la nube · 1

De stdio a un servidor HTTP

Hoy weather-mcp corre por stdio: el host lo lanza como subproceso local y le habla por entrada/salida estándar. Por diseño, eso solo funciona en la misma máquina. Para que otra persona lo use sin clonar el repo, el servidor debe escuchar en un puerto de red — el transporte streamable-http del SDK de MCP.

Idea clave

“stdio conecta procesos dentro de un mismo computador; streamable-http conecta cualquier cliente MCP a través de una URL, igual que una API REST.”

stdio · como está hoy

  • El host ejecuta uv run weather-mcp como subproceso propio.
  • Solo quien tiene el repo clonado en su máquina puede usarlo.
  • No hay URL ni puerto — no hay nada que "exponer" a internet.

streamable-http · para la nube

  • El proceso escucha en un puerto, por ejemplo 0.0.0.0:8000.
  • Cualquier cliente MCP se conecta con una URL: https://tu-dominio.com/mcp.
  • Un solo servidor desplegado atiende a muchos usuarios a la vez.
Seguridad en la nube

Proteger el MCP de código malicioso

Exponer un servidor a internet sin protección permite que cualquiera ejecute tus herramientas. Imagina si tu MCP tiene acceso a bases de datos o borra archivos. Necesitas una capa de autorización robusta.

Autenticación y AutorizaciónNo basta con saber quién eres, el servidor debe verificar si tienes permisos para ejecutar la herramienta.
OAuth 2.1El estándar moderno recomendado para proteger servidores MCP y validar tokens seguros (JWT).
Validar el Token en FastAPIUsando una dependencia HTTP para rechazar conexiones anónimas y prevenir ataques.
Cómo requiere conexión remota seguraClaude Code
# Claude debe incluir un Bearer Token generado 
# por un servidor de Autorización.
claude mcp add --transport http secure-mcp \
  https://tu-dominio.com/sse \
  --header "Authorization: Bearer eyJhbGciOiJ..."

# Si el token falta o es inválido, el servidor MCP 
# devolverá un 401 Unauthorized y no ejecutará nada.
Workflow OAuth 2.1

El flujo con Scalekit

Implementar OAuth desde cero es complejo. Scalekit simplifica la validación de tokens y descubrimiento. Este es el flujo de trabajo (Workflow) cuando un Agente/Cliente se conecta.

🤖
Paso 1Cliente MCP

El LLM o Agente solicita acceso y necesita un Token para conectarse al servidor.

🔐
Paso 2Auth Server

Scalekit emite un Token JWT (Bearer) tras autenticar correctamente al cliente.

📩
Paso 3Petición SSE

El cliente llama al endpoint HTTP (/sse) del Servidor MCP enviando el Token.

🛡️
Paso 4Validación

El Servidor MCP usa scalekit-sdk-python para verificar la firma y los permisos.

✅
Paso 5Acceso Ok

El MCP aprueba la conexión y permite ejecutar las Herramientas (Tools) seguras.

Dato clave de Seguridad: El servidor MCP jamás te pide usuario y contraseña directamente. El formulario de inicio de sesión vive exclusivamente en Scalekit, permitiéndote ofrecer logins con Gmail, GitHub o Correos Institucionales sin escribir una sola línea de código en Python.
